# 포즈 작업기억 게임 개발 계획

## 1. 목표 요약

- 브라우저에서 실행되는 포즈 기반 작업기억 게임을 개발합니다.
- 사용자는 화면에 제시되는 동작 시퀀스를 기억한 뒤 같은 순서로 재현합니다.
- 포즈 추론, 동작 판정, 순서 비교는 모두 브라우저에서 처리합니다.
- 백엔드는 게임 결과 기록과 사용자별 이력 조회를 담당합니다.
- 게임에 사용하는 시각, 음향, 안내용 리소스는 공개 라이선스의 오픈 리소스를 우선 사용합니다.

## 2. 요구사항 정리

### 게임 핵심 규칙

- 동작은 클래스 번호로 관리합니다.
- 동작 클래스는 다음과 같습니다.
  - `0`: 기본 자세
  - `1`: 양팔 들기
  - `2`: 왼팔 들기
  - `3`: 오른팔 들기
  - `4`: 스쿼트
  - `5`: 왼쪽 기울기
  - `6`: 오른쪽 기울기
  - `7`: 점프
- 정답 시퀀스와 사용자 시퀀스가 순서까지 모두 일치하면 성공입니다.
- 게임 난이도는 1단계부터 10단계까지 제공합니다.

### 프론트엔드 요구사항

- 브라우저에서 바로 플레이할 수 있어야 합니다.
- 로그인 없이 게임 플레이와 단계별 스코어 확인이 가능해야 합니다.
- 제시된 동작과 사용자의 현재 동작을 함께 보여줘야 합니다.
- 포즈 일치 여부와 시퀀스 진행 상태를 시각적으로 보여줘야 합니다.
- 정답 여부는 애니메이션 효과로 명확하게 피드백해야 합니다.
- MediaPipe 기반 추론과 판정은 브라우저에서 수행해야 합니다.

### 백엔드 요구사항

- FastAPI로 구성합니다.
- PostgreSQL에 게임 성공/실패 기록을 저장합니다.
- DB는 Docker 환경을 사용합니다.
- 사용자는 비밀번호 없이 아이디만으로 로그인합니다.
- 사용자는 마이페이지에서 자신의 게임 기록을 조회할 수 있어야 합니다.

### 리소스 사용 요구사항

- 게임에 사용하는 리소스는 오픈되어 있는 리소스를 사용합니다.
- 라이선스가 불명확한 리소스는 사용하지 않습니다.
- 상업적 사용 가능 여부와 출처 표시 의무를 사전에 확인합니다.
- 최종 적용 리소스는 라이선스 정보를 문서로 함께 정리합니다.

## 5. 구현 전략

### 1단계: 프로젝트 골격 구성

- 프론트엔드와 백엔드 디렉터리 구조를 분리합니다.
- FastAPI 기본 서버와 헬스체크 API를 구성합니다.
- PostgreSQL Docker 구성을 추가합니다.
- 프론트엔드 개발 환경과 기본 라우팅을 구성합니다.

### 2단계: 인증과 기본 데이터 구조

- 아이디만 입력받는 간편 로그인 흐름을 구현합니다.
- 사용자, 게임기록 테이블을 설계합니다.
- 로그인 상태를 프론트엔드에서 유지할 최소 구조를 만듭니다.

### 3단계: 게임 규칙 엔진 구현

- 단계별 시퀀스 길이 증가 규칙을 정의합니다.
- 동작 클래스 목록과 시퀀스 비교 로직을 구현합니다.
- 성공/실패 판정 규칙을 모듈화합니다.

### 4단계: 포즈 인식과 동작 분류

- 브라우저에서 웹캠 입력을 받습니다.
- MediaPipe로 관절 좌표를 추출합니다.
- 포즈 추론 목표 성능은 기본 `12 FPS`로 설정합니다.
- 저사양 환경 대응 최소 기준은 `10 FPS`로 보고, `5 FPS`는 비권장 기준으로 분류합니다.
- 각 동작을 판정하는 규칙 기반 분류기를 구현합니다.
- 기본 자세와 실제 입력 동작 사이의 전환 안정화 로직을 추가합니다.

### 5단계: 게임 플레이 화면 구현

- 목표 동작 시퀀스를 단계별로 애니메이션과 함께 보여줍니다.
- 현재 사용자의 포즈와 인식 결과를 실시간으로 표시합니다.
- 사용자가 수행한 동작 시퀀스를 화면에 누적 표시합니다.
- 정답, 오답, 대기 상태에 따라 시각 효과를 다르게 제공합니다.

### 6단계: 결과 저장과 마이페이지 구현

- 성공/실패 결과를 백엔드에 저장합니다.
- 사용자별 게임 기록 조회 API를 연결합니다.
- 마이페이지에서 최근 기록, 단계, 성공 여부를 확인할 수 있게 합니다.

### 7단계: 오픈 리소스 적용과 문서화

- 아이콘, 효과음, 배경 그래픽, 안내용 일러스트가 필요하면 오픈 리소스만 사용합니다.
- 리소스 후보를 선정할 때 라이선스 조건을 비교합니다.
- 사용 확정 리소스의 출처와 라이선스를 별도 문서로 정리합니다.

### 8단계: 안정화와 품질 보완

- 포즈 판정 오탐과 중복 입력을 줄이는 디바운스 또는 쿨다운 로직을 넣습니다.
- 웹캠 권한 거부, 카메라 미연결, 저조도 환경에 대한 예외 처리를 추가합니다.
- 모바일/데스크톱 브라우저 기본 대응 범위를 점검합니다.

## 6. 세부 설계 포인트

### 동작 판정 방식

- 관절 각도와 상대 좌표를 기반으로 규칙을 정의합니다.
- 추론 주기는 기본 `12 FPS`를 목표로 하며, 반응성과 브라우저 부하의 균형 기준으로 사용합니다.
- `5 FPS`는 정적인 자세 확인에는 가능하지만 점프나 빠른 전환 동작 누락 가능성이 커서 기본 운영값으로는 사용하지 않습니다.
- 예시
  - 양팔 들기: 양 손목이 양 어깨보다 충분히 위에 있음
  - 오른팔 들기: 오른 손목만 오른 어깨보다 위에 있음
  - 스쿼트: 무릎 각도와 골반 높이가 기준 이하로 내려감
  - 왼쪽 기울기: 어깨선 또는 몸통 중심축이 왼쪽으로 기울어짐
  - 점프: 발 또는 골반 중심의 순간적인 수직 이동 감지

### 시퀀스 입력 안정화

- 한 프레임 판정만으로 입력하지 않고 일정 시간 유지 시 확정합니다.
- 권장 확정 조건은 `0.3~0.5초` 유지이며, `12 FPS` 기준으로 약 `4~6프레임` 연속 감지를 기준값으로 삼습니다.
- 기본 자세를 사이에 두어 다음 동작 입력을 받는 방식이 안전합니다.
- 동일 동작 연속 중복 입력 방지 규칙이 필요합니다.

### 성능 기준

- 기본 목표: 브라우저 포즈 추론 `12 FPS`
- 최소 허용: `10 FPS`
- 비권장: `5 FPS`
- 성능 저하 시에는 해상도 축소, 추론 간격 조절, UI 효과 경량화를 우선 적용합니다.
- FPS가 기준 이하로 오래 떨어지면 사용자에게 카메라 환경 또는 기기 성능 안내를 제공합니다.

### 난이도 설계

- 1단계는 1개 동작부터 시작합니다.
- 단계가 올라갈수록 시퀀스 길이와 제시 속도를 조절합니다.
- 10단계 구성 예시
  - 초반: 1~3개 동작
  - 중반: 4~6개 동작
  - 후반: 7개 이상 동작과 빠른 제시

### 오픈 리소스 적용 원칙

- 가능하면 `CC0`, `MIT`, `Apache-2.0`, 상업 사용 가능한 `CC BY` 계열 리소스를 우선 검토합니다.
- `CC BY` 리소스는 필요한 출처 표기 위치를 제품 내 또는 문서로 확보합니다.
- `CC BY-SA`는 파생물 공개 의무 범위를 검토한 뒤 사용 여부를 결정합니다.
- 비상업용 전용 라이선스는 배포 계획에 따라 제외하는 쪽을 우선합니다.

## 8. API 초안

### 인증

- `POST /auth/login`
  - 입력: `user_id`
  - 처리: 없으면 생성, 있으면 그대로 로그인

### 게임 기록

- `POST /games/results`
  - 입력: `user_id`, `level`, `target_sequence`, `player_sequence`, `is_success`, `score`
  - 처리: 플레이 결과 저장

- `GET /games/me/results`
  - 입력: 사용자 식별값
  - 처리: 사용자 기록 목록 반환

## 9. 검증 계획

### 기능 검증

- 동작별 분류가 의도대로 동작하는지 확인합니다.
- 정답 시퀀스와 사용자 시퀀스 비교가 순서까지 정확히 반영되는지 확인합니다.
- 성공/실패 결과가 DB에 정상 저장되는지 확인합니다.
- 마이페이지에서 사용자 본인 기록만 조회되는지 확인합니다.
- 기본 환경에서 포즈 추론이 `12 FPS` 전후로 유지되는지 확인합니다.
- 저사양 환경에서도 최소 `10 FPS` 수준으로 동작 가능한지 확인합니다.

### 리소스 검증

- 사용한 이미지, 아이콘, 사운드의 출처가 모두 확인되는지 점검합니다.
- 라이선스 문구와 출처 표기 요건이 누락되지 않았는지 확인합니다.
- 배포 환경에서 라이선스 충돌 가능성이 없는지 검토합니다.

### 사용자 경험 검증

- 제시 동작이 충분히 알아보기 쉬운지 확인합니다.
- 일치 여부 피드백이 즉시 전달되는지 확인합니다.
- 카메라 사용 흐름이 복잡하지 않은지 확인합니다.

### 예외 상황 검증

- 카메라 권한 거부
- 포즈 미검출
- 네트워크 오류
- 빈 사용자 ID 입력
- 브라우저 성능 저하

## 12. 확인이 필요한 가정

- 프론트엔드 기술은 아직 명시되지 않았으므로 React 또는 유사 SPA 구조를 가정할 수 있습니다.
- 로그인은 보안 인증보다 사용자 식별 목적의 간편 로그인으로 해석했습니다.
- 스코어 계산 방식은 요구사항에 명시되지 않아 성공/실패와 단계 중심으로 우선 설계했습니다.
- 동작 제시 방식은 정적 이미지보다 애니메이션 또는 안내 오버레이 중심이 적합하다고 판단했습니다.
- 오픈 리소스는 재배포와 상업적 사용 가능 범위를 우선 고려해 선정하는 것으로 가정했습니다.
